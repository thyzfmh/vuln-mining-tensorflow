# AI Vulnerability Mining Competition Package

This repository is packaged for one task: use OpenCode to discover real,
runtime-verified vulnerabilities in the judge-provided source tree.

## Entry

Use the repository root entry file:

```text
INSTRUCTION.md
```

It tells OpenCode to load and execute:

```text
work/skills/vuln-mining-autonomous/SKILL.md
```

## Fixed Task

- Target source: the entry orchestrator automatically discovers it from the judge asset root and persists only an internal path context under `work/`
- Work output root: all runtime artifacts beneath `work/`
- Skill name: `vuln-mining-autonomous`
- Entry orchestrator: `python3 work/run_vulnerability_mining.py`
- Final gate: `python3 work/skills/vuln-mining-autonomous/scripts/final_verify.py --work-root work`
- Required deliverables: `work/vulnerability_list.md`, `work/llm_chat_log.json`, `work/vulnerability_report.md`, `work/verify/run_test.py`
- Required method evidence: `work/reports/source-file-manifest.md`, `work/reports/toolchain-capabilities.md`, `work/reports/verification-escalation.md`, `work/reports/coverage-ledger.md`, `work/reports/scan-completion.md`, one or more `work/plans/scan-wave-NNN.md` files

The skill owns the full loop: target discovery, inventory, deterministic SAST
candidates, black-box LLM review logging, AI-generated tests, runtime
verification, verification-tool probing and escalation, candidate-space
exhaustion, coverage-ledger maintenance, final reports, and result update.

## Platform Asset Discovery

The platform asset discovery entry orchestrator first scans `/app/code/judge-assets/01_02_vulnerability_detection`, locates the source tree, and writes its resolved path to `work/.vuln-mining-target-root`. Every phase uses that internal context, so platform execution needs no manual environment setup.

For explicit local tests, the package also resolves a target non-interactively:

1. `--target-root` CLI argument (highest priority)
2. `VULN_TARGET_ROOT` environment variable
3. `TARGET_ROOT` environment variable
4. `<repo>/code` if it directly contains source files
5. A single non-hidden subdirectory under `<repo>/code` that contains source

The work output root is resolved similarly via `--work-root`,
`VULN_WORK_ROOT`, `WORK_ROOT`, defaulting to `<repo>/work`.

Runtime deliverables and reports are generated by the skill beneath `work/` and
replace tracked `BOOTSTRAP_PENDING` files during execution. Tracking these
initial files is required because the judge reads key paths from the submitted
read-only package; the final gate still rejects them until real evidence exists.

all production subprocesses run non-interactively with workspace-local scratch
files; no human authorization step is part of the execution contract.
