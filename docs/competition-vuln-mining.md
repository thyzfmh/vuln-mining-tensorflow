# AI Vulnerability Mining Competition — TensorFlow

This branch adds a vulnerability mining profile for AI-driven security auditing of a large C++ codebase.

The goal is to demonstrate an engineering methodology for using LLMs to discover real vulnerabilities in production code, with complete evidence chains and AI-generated proof-of-concepts.

## Competition Requirements

1. **Black-box standard**: Never tell the AI the project name, version, or known bugs
2. **Complete interaction log**: All LLM turns recorded without editing, enabling 90% reproducibility
3. **AI-generated tests**: Any PoC or test must be generated by the AI, not hand-written
4. **Evidence-based**: Every vulnerability must have source code evidence and a trigger path
5. **Engineering methodology**: Systematic scanning combining traditional (SAST/SCA) and LLM approaches

## Deliverables

| File | Description |
|------|-------------|
| `result/vulnerability_list.md` | Complete vulnerability list with evidence |
| `result/llm_chat_log.json` | Complete LLM interaction log |
| `result/vulnerability_report.md` | Detailed vulnerability audit report |

## Usage

### Platform Evaluation

Use the packaged entry:

```text
INSTRUCTION.md
work/skills/vuln-mining/SKILL.md
```

### Local Setup

Prepare the target code:

```bash
cd code
git clone https://github.com/tensorflow/tensorflow
cd tensorflow
git checkout v2.11.0
```

Run the harness:

```bash
./work/skills/vuln_mining_tf_blackbox/scripts/analyze_target.sh ./code/tensorflow
./work/skills/vuln_mining_tf_blackbox/scripts/scan_sast.sh ./code/tensorflow
./work/skills/vuln_mining_tf_blackbox/scripts/scan_llm.sh ./code/tensorflow
./work/skills/vuln_mining_tf_blackbox/scripts/verify_vulnerabilities.sh
./work/skills/vuln_mining_tf_blackbox/scripts/final_verify.sh
```

### OpenCode Loop

```text
read scan plan
  -> craft black-box compliant prompt (no version/CVE hints)
  -> send prompt to LLM and record interaction
  -> verify evidence for each candidate vulnerability
  -> run anti-hallucination checklist
  -> if evidence fails, refine prompt and re-scan
  -> generate PoC for confirmed vulnerabilities
  -> update deliverables (vulnerability_list.md, llm_chat_log.json, vulnerability_report.md)
  -> continue with the next wave only after verification
```

## Design Philosophy

This profile intentionally favors systematic, evidence-based vulnerability discovery over brute-force scanning:

- Black-box compliance ensures the methodology is reproducible
- Multi-layer scanning (SAST + LLM + evidence) catches different bug classes
- Anti-hallucination checks prevent false positives
- AI-generated PoCs prove vulnerability triggerability
- The engineering process itself is the competition deliverable

## Scoring Criteria

The competition evaluates:

1. **Bug discovery quality** — real, triggerable vulnerabilities with evidence
2. **Engineering methodology** — systematic process, not random probing
3. **LLM utilization** — how effectively LLM capabilities are leveraged
4. **Traditional + innovative methods** — combining SAST/SCA with LLM analysis
5. **Reproducibility** — 90% reproducibility from interaction logs
